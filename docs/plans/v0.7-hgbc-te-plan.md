# v0.7 - HistGradientBoosting + Exact-Value Target Encoding Plan (Rung 7)

Goal: test whether `HistGradientBoostingClassifier` (sklearn's native GBM, a
4th distinct tree-boosting implementation beyond LightGBM/CatBoost/XGBoost) plus
exact-value target encoding of the 7 numeric features beats our current best
(CatBoost-V1, OOF 0.9493). Surfaced by investigating
`redamountassir/ps-s6e7-hgbc-baseline-lb-0-95034-cv-0-95026` ("TE-HGBC") — see
`docs/investigate/2026-07-03-kaggle-discussion-findings.md`.

**Run venue: Kaggle**, not local JupyterLab — build the notebook with
Kaggle-input-first data loading from the start (matching the convention used when
publishing v0.1-v0.6), smoke-test locally on a data sample first, then push and
run on Kaggle's own compute.

## What's genuinely new here (vs. v0.1-v0.6)

1. **Exact-value target encoding of numeric features.** All 13 raw columns
   (6 categoricals + 7 numerics) target-encoded via sklearn's native
   `TargetEncoder(cv=5, target_type='multiclass')`, with numerics treated as
   exact-value categories (cast to string) rather than binned/qcut'd. Rationale
   (from the source notebook): with 690k rows, each distinct numeric value
   repeats hundreds of times, so per-value label rates directly estimate the
   generator's probability — signal `HistGradientBoostingClassifier`'s ≤255
   histogram bins can't fully resolve on their own (neighboring values get
   merged). This is different from our own v0.2/v0.3-V2 engineered feature set,
   which only target-encodes specific categorical/cross columns, never raw
   numerics at exact-value granularity.
2. **`HistGradientBoostingClassifier`** with native `class_weight='balanced'`
   (simpler than the source notebook's manual `sample_weight` helper — sklearn
   1.9 supports `class_weight` directly) and native categorical handling via
   `categorical_features='from_dtype'`.

## Scope decisions

- Reuse the source notebook's tuned hyperparameters as-is (specific float values
  suggest a real hyperparameter search was already done) — this experiment is
  about the TE+HGBC combination, not re-tuning capacity, matching the project's
  standing "combine existing best efforts, don't re-tune" philosophy for
  cross-notebook techniques.
- Use our own established single 5-fold `StratifiedKFold(random_state=42)` split
  (not the source notebook's 3-seed x 5-fold = 15-fit scheme) for consistency
  with the rest of this project and faster turnaround on Kaggle's shared compute.
- No post-hoc prior/threshold correction — plain argmax, matching both this
  technique's source notebook and our own CatBoost models' single-correction
  discipline (training-time class weighting only).

## Approach

1. Load data with Kaggle-input-first resolution + local fallback.
2. Build two views of the 13 raw features per fold:
   - **Model view**: native `pandas.Categorical` dtype for the 6 categoricals
     (shared categories, NaN as its own level), numerics left as-is (HGBC
     natively routes NaN, no imputation needed).
   - **TE view**: all 13 columns cast to string (numerics at exact value,
     categoricals unchanged), NaN filled with a placeholder.
3. Per outer fold: fit `TargetEncoder(cv=5, smooth='auto', target_type='multiclass')`
   on the TE view of that fold's training rows only (leak-free via internal
   cross-fitting), transform validation/test; concatenate model-view + the
   resulting TE columns as the final feature frame.
4. Train `HistGradientBoostingClassifier(class_weight='balanced',
   categorical_features='from_dtype', early_stopping=True, **source notebook's
   tuned config)`. Capture OOF + test probabilities.
5. Compare solo OOF against the known CatBoost-V1 (0.9493) and the source
   notebook's own reported CV (0.95026) as a sanity check.
6. Reproduce CatBoost-V1 alongside for a blend check + nested validation, same
   honest-improvement discipline as Rung 3/4/6.
7. Submit only if nested validation confirms a real improvement over the current
   best (~0.9493 OOF / ~0.94885-0.94937 LB).

## Acceptance

- Smoke-tested locally on a data sample before pushing to Kaggle (catch API/data
  issues before spending Kaggle compute time).
- Any blend improvement must be nested-validated, not a same-data optimistic fit.
- Findings recorded in `leaderboard.md`, `notebook-runs.md`, `implementation-plan.md`
  regardless of outcome.
