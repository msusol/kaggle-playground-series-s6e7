# v0.8 RealMLP Implementation Plan

## Goal

Test whether a RealMLP (neural-net) architecture adds real, non-redundant signal
vs. our current best model (v0.7 HGBC-TE, OOF 0.9502), either solo or as a blend
member. Run locally first (Apple M3 Pro, PyTorch MPS backend) before deciding
whether to push to Kaggle.

## Context

- Surfaced by investigating `yunsuxiaozi/pss6e7-realmlp-cv-0-95063` — see
  `docs/investigate/2026-07-05-kaggle-discussion-findings.md`. Raw CV 0.95057
  (before its own post-hoc correction), a genuinely different model family
  (neural net vs. all-tree-boosting so far) with a comparable score to v0.7.
- This is the first candidate diversity source for blending since v0.5 (all
  prior blend attempts were tree-boosting variants too correlated with
  CatBoost to help).

## What's genuinely new vs. v0.1-v0.7

- First neural-net architecture in this project (all prior work: LightGBM,
  CatBoost, XGBoost, HistGradientBoosting — all tree-boosting).
- Periodic/Fourier-style numeric feature embeddings (`PBLDEmbedding`), NTK-style
  linear layers, a 16-way "ensemble baked into one model" trick, EMA of weights.

## Scope decisions

- **Fold split**: use our own `StratifiedKFold(5, shuffle=True, random_state=42)`
  (matching every other version in this project), not the source notebook's
  naive `index % num_folds` assignment.
- **Device**: adapt `torch.device("cuda" if ... )` to also check
  `torch.backends.mps.is_available()` for local Apple Silicon runs.
- **Architecture**: kept faithful to the source (PBLD embeddings, NTPLinear,
  categorical layer, RankGLU head, `n_ens=16`, EMA, 5-param-group LR schedule) —
  the point is testing whether the architecture itself is competitive, not a
  simplified version.
- **Single correction only**: the architecture's own training-time
  `compute_class_weight('balanced')` (with its fixed `[0.9, 1.1, 1.0]` tweak) is
  kept as-is. The source notebook's post-hoc Optuna per-class reweighting is
  **not reproduced** — our own investigation already found it adds only
  +0.00006 (negligible), consistent with 5 prior independent findings that a
  tunable second correction on top of balanced training finds ~nothing extra.
  Plain argmax, per our established discipline.
- **Comparison peg**: reproduce CatBoost-V1 (v0.3's exact config) for a blend
  check + nested validation, same pattern as every prior version.
  `CURRENT_BEST_OOF = 0.9502` (v0.7, our current best — not CatBoost-V1's
  0.9493, since v0.7 superseded it).
- **Local run first**: build and smoke-test locally (data at `../data`,
  `.venv` at the workspace root) before considering a Kaggle push. No
  Kaggle-input-first path needed yet for this local-only phase.

## Approach

1. Adapt the pulled notebook's PyTorch code: MPS device detection, our own
   StratifiedKFold, drop the post-hoc Optuna correction, add the standard
   CatBoost-V1 reproduction + blend/nested-validation cells matching v0.5/v0.6/v0.7's
   pattern.
2. Smoke-test on a reduced sample (fewer rows, fewer epochs) to confirm the
   architecture runs correctly on MPS before committing to a full run.
3. Full local run: solo RealMLP OOF, CatBoost-V1 reproduction check, blend
   weight search, nested validation, decision logic (submit only if nested
   validation clears `CURRENT_BEST_OOF + 0.0005`).
4. Record result in `leaderboard.md` regardless of outcome (positive or
   negative, per project discipline).
5. Only after a satisfactory local result, consider whether to push to Kaggle
   (would need `enable_gpu: true`, `machine_shape: NvidiaTeslaT4` — a first for
   this project's kernel-metadata.json files).

## Acceptance criteria

- Notebook runs end-to-end locally without error.
- OOF balanced accuracy reported for RealMLP solo, CatBoost-V1 reproduction
  PASS/FAIL, blend same-data result, nested-validated honest improvement.
- Result recorded in `leaderboard.md` whether positive, negative, or flat.
