# v0.6 - XGBoost One-vs-Rest Plan (Phase 6)

Goal: test whether a one-vs-rest (OvR) problem formulation — 3 independent binary
classifiers, one per class, combined via argmax — beats native multiclass on this
dataset. Surfaced by discussion 718258 (Masaya Kawamata), where `XGB_OvR` scored
highest of 13 model families (CV 0.95036 / LB 0.95040), modestly above native
multiclass XGBoost variants. See
`docs/investigate/2026-07-03-kaggle-discussion-findings.md` for the full research.

Given the repeated, now heavily-corroborated synthesis-noise-ceiling finding (Rung
3, Rung 4, and this same discussion thread's 13-model-family sweep), expected
upside here is small (~0.0003-0.0005, within that thread's own stated noise band).
Worth trying since it's a genuinely untried formulation, not a rehash of blend
weights or thresholds — but budget effort accordingly.

## Scope decisions (confirmed with user)

- **Real XGBoost**, not CatBoost's native `MultiClassOneVsAll` loss — matches
  exactly what the discussion tested, at the cost of adding `xgboost` as a new
  project dependency and building a new training harness (no reuse of the
  LightGBM/CatBoost CV harness patterns beyond structure).
- **Engineered features** (v0.2/v0.3-V2's 35-feature set), not the 13 base
  features — richer feature set, consistent with wanting the strongest possible
  comparison point.
- Keep our own established 5-fold `StratifiedKFold(random_state=42)` split (not
  Kawamata's 7-fold) for direct comparability with our own leaderboard history.

## Approach

1. Reconstruct the 35-feature engineered set (same code as v0.4/v0.5 — missingness
   indicators, categorical crosses, OOF smoothed target encoding).
2. For each of 5 folds: train 3 independent binary XGBoost classifiers (one per
   class, `binary:logistic`, `scale_pos_weight` for class imbalance — XGBoost's
   analogue to `class_weight='balanced'`), native categorical handling via
   `enable_categorical=True` + shared `pd.Categorical` dtype (same approach as our
   LightGBM setup). Each binary classifier early-stops on its own binary metric
   (AUC) — there's no single early-stopping signal that corresponds to the final
   combined multiclass balanced accuracy in a true OvR setup, worth being
   transparent about as a structural limitation of this formulation.
3. Combine: argmax over the 3 raw per-class probabilities (standard OvR
   combination), capturing OOF + test probabilities.
4. Reproduce CatBoost-V1 (base features, our current best single model) alongside,
   for a same-data comparison peg and an optional blend check.
5. Solo comparison: XGB-OvR vs. CatBoost-V1 (0.9493) vs. known v0.3-V2 (0.9491).
6. If XGB-OvR is competitive, try a 2-way blend (XGB-OvR + CatBoost-V1) + nested
   validation, same honest-estimate discipline as Rung 3/4.
7. Submit only if nested validation confirms a real improvement over the current
   best (~0.9493 OOF / ~0.94885-0.94913 LB).

## Acceptance

- Report the solo XGB-OvR result honestly regardless of outcome (positive or
  negative — both are informative, matching this project's standing practice).
- Any blend improvement must be nested-validated, not a same-data optimistic fit.
- Findings recorded in `leaderboard.md`, `notebook-runs.md`, `implementation-plan.md`
  regardless of outcome.
